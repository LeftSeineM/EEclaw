# 网络学堂 vs 信息门户：登录逻辑与爬取流程

## 一、网络学堂（learn.tsinghua.edu.cn）

### 1.1 登录流程

```
用户点击登录 → auth:login → loginWithCredentialsAsync
    ↓
casLogin.loginProgrammatic(username, password, 'learn')
    ↓
getCasFromLearn(): 访问 learn.tsinghua.edu.cn/f/login
    → 直接返回 HTML，内含 window.location.href = 'CAS_URL'
    → extractCasUrl() 解析出 CAS 地址
    ↓
访问 CAS 页 → 解析表单 → SM2 加密密码 → POST 提交
    ↓
跟随重定向 → 获取 cookies（learn、id 等）
    ↓
persistAuthState(cookies) → 写入 auth-session.json
    ↓
登录成功后立刻 crawler.fetchAll() 拉取网络学堂数据
```

**特点：**

- **CAS 获取简单**：`learn/f/login` 直接返回带 CAS 跳转的页面，`extractCasUrl` 即可解析
- **纯 HTTP 流程**：程序化登录成功率高，无需 BrowserWindow
- **爬取简单**：`crawler.js` 用 Node.js `https` + `auth.getCookieHeaderForCrawler()` 直接请求 learn API
- **一次登录即可用**：主界面登录后立刻能刷新课程、公告、作业

## 二、信息门户（info.tsinghua.edu.cn）

### 2.1 登录流程（按需触发）

信息门户**不在主登录时**登录，只在用户使用「培养方案」等功能时按需登录：

```
用户点击「培养方案」→ courseSelectionCrawler.fetchTrainingPlanHTML
    ↓
检查 hasValidInfoSession() → 若已有 info cookies 则跳过
    ↓
若无会话且 hasCredentials()：
    ensureLoginToInfo(cred) → casLogin.loginProgrammatic(..., 'info')
        ↓
    getCasFromInfo(): 访问 info.tsinghua.edu.cn/
        → info 首页不会直接重定向到 CAS
        → 需解析 HTML：extractCasUrl、onload、class="onload"、302 等
        → 比 learn 复杂，解析失败率更高
    ↓
若程序化失败：继续 BrowserWindow 流程
    ↓
打开 BrowserWindow → 加载 info 首页
    ↓
检测 needLogin（右上角「登录」）→ 点击或直接 loadURL(casUrl)
    ↓
在 CAS 页：getAuthInjectScript 自动填充并提交（需等待 3 秒渲染）
    ↓
登录成功后：打开 yyfwsearch 搜索「培养方案完成情况」
    ↓
点击「应用导航」→ 查找并点击「培养方案完成情况」卡片
    ↓
进入培养方案页 → 提取 HTML
```

### 2.2 爬取流程

- **依赖 BrowserWindow**：培养方案需在真实浏览器中完成「搜索 → 点击应用 → 进入页面」
- **无法纯 HTTP**：info 应用入口、培养方案页面依赖 JS 渲染和会话
- **步骤多**：加载 info → 可能 CAS 登录 → 搜索页 → 点击应用导航 → 点击卡片 → 提取

## 三、差异对比与「不顺畅」原因

| 维度 | 网络学堂 | 信息门户 |
|------|----------|----------|
| **主登录是否包含** | ✅ 是，主登录即登录 learn | ❌ 否，主登录只登录 learn |
| **CAS 获取** | learn/f/login 直接返回 CAS URL | info 首页需解析 HTML/302，更复杂 |
| **程序化登录成功率** | 高 | 较低，常需 fallback 到 BrowserWindow |
| **爬取方式** | 纯 Node.js HTTP + cookies | 必须 BrowserWindow 模拟点击 |
| **触发时机** | 登录后立刻 crawler.fetchAll | 用户点「培养方案」时才按需登录+爬取 |
| **用户感知** | 一次登录，刷新即可 | 需先保存凭据，再用培养方案时可能弹窗、等待 |

**信息门户不顺畅的主要原因：**

1. **主登录只登 learn**：用户在主界面登录后，info 并未登录，用培养方案时才临时登录，容易失败或弹窗
2. **info 获取 CAS 更脆弱**：info 首页结构可能变化，`extractCasUrl`、`onload` 等选择器易失效
3. **流程长且依赖 UI**：搜索 → 应用导航 → 点击卡片，任一步 DOM 变化都会失败
4. **LoginModal 文案**：写「登录信息门户」，但实际主登录目标是 learn，容易混淆

## 四、调整建议

### 4.1 主登录时同时登录 learn + info（推荐）

在主登录成功后，若用户勾选了「保存凭据」，**顺带调用 `ensureLoginToInfo`**，这样：

- 一次登录即可同时获得 learn 和 info 会话
- 用户后续用培养方案时，`hasValidInfoSession()` 已为 true，无需再弹窗登录

**修改位置**：`electron/main.js` 中 `auth:login` 成功后：

```javascript
if (!result.success) return result;
if (saveCredentials) {
  auth.saveCredentials(username.trim(), password);
}
// 新增：同时尝试登录 info，便于培养方案等功能
if (saveCredentials) {
  const cred = { username: username.trim(), password };
  auth.ensureLoginToInfo(cred).catch(() => {}); // 失败不阻塞主流程
}
// 原有：拉取网络学堂
try {
  const fetchResult = await crawler.fetchAll();
  ...
}
```

### 4.2 增强 info 的 CAS 获取

在 `casLogin.js` 的 `getCasFromInfo` 中增加更多 fallback：

- 优先检查 302 的 `location` 是否包含 `id.tsinghua.edu.cn`
- 增加更多 HTML 选择器（如 `a[href*="id.tsinghua"]`、`a[href*="id.sigs"]`）
- 若 info 首页仍无法解析，可 fallback 到 `getCasFromLearn()`（因为 CAS 是统一的）

### 4.3 统一 LoginModal 文案

将「登录信息门户」改为更准确的描述，例如：

- 「登录清华大学网络学堂与信息门户」
- 或「登录后可使用网络学堂与培养方案（选课）」

### 4.4 培养方案入口提前提示

在「培养方案」入口处，若检测到 `!hasValidInfoSession() && !hasCredentials()`，可提前提示：

「使用培养方案需先登录并勾选「保存凭据」，请先在主界面完成登录。」

避免用户点进培养方案后才报错。
